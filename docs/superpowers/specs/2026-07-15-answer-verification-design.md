### 0-1. 사전 발견 — 한창민 정당 오표기는 문자열 대조로 잡힌다 (QA 검증 반영, 초안 정정)

초안은 `party.py` 의 `SATELLITE_PARENT` 병합 때문에 주입값과 출력이 같아 검출
불가라고 판단했으나 **사실과 반대다**(QA 실측 확인). `SATELLITE_PARENT`
(party.py:40-43)는 **여야 판정 전용**이고 표기(라벨)는 위성정당 원본을 유지한다
(2026-07-03 사용자 결정, 주석 명시). eval_057 의 source_block 실물도 한창민을
`[더불어민주연합(당시 여당)]` 으로 주입했는데 LLM 답변은 "한창민 위원(더불어민주당)"
— **주입값 ≠ 출력**이다. 즉 이 오표기는 답변 속 정당명을 인용 근거의 주입 라벨과
문자열 포함 비교하는 것만으로 잡히는 화자/근거 불일치이며, §4-2 에 **정당 라벨
일치** 규칙으로 정식 포함한다(§6 `party_label_consistency`). 범위 밖이 아니다.

## 1. 확장 대상인가 별도 층인가 — 설계 결정

`backend/grounding.py` 의 `judge()`는 **문서 단위**(인용 유무 + 거절 문구 유무)
판정만 한다. 문장 단위 대조는 판정 로직이 아니라 **새로운 분석 단계**(문장 분리,
근거 매핑, 부분 검증)이므로 grounding.py 를 확장하지 않고 **신규 모듈
`backend/verification.py`** 를 추가한다(POL-8 의 `issue_context.py` 분리 패턴과
동일 — answer.py·grounding.py 비대화 방지 + 책임 분리).

- `grounding.judge()` 는 무변경. 대신 `verification.py` 의 결과(§2~4 의 flag 목록)를
  **문서 등급의 새 강등 사유 하나**로만 접합한다: `invalid_citations` 가 있으면
  `FULL→PARTIAL` 강등하는 기존 규칙과 동일한 자리에, `verification_flags` 가
  비어있지 않으면 마찬가지로 강등. **새 등급 신설은 하지 않는다**(FULL/PARTIAL/
  REFUSED/NONE 4단계 유지 — 마스터 4-9 계약 존중).
- `verify()` 는 **`generate_answer()` 내부**에서 호출되어(위치 근거는 §6 —
  `_source_summary` 200자 절단 전의 근거 전문이 필요) 반환 dict 에 `verification`
  키로 실린다. `main.py` 는 그 값을 읽어 `judge()` 결과에 강등만 접합하고,
  `query_logs` 에 원본 그대로 적재(POL-8 의 `issue_context` 저장 패턴 재사용).

## 2. 비교 질문 전용 가드

### 2-1. 진영 커버리지 (핵심 — eval_068, eval_019 의 일부)

기존 `_TYPE_GUIDES["compare"]`(생성 전 프롬프트)와 `issue_context_for(style="qa")`
(POL-8 확장, 2026-07-14)는 이미 "소수 발언 일반화 금지"를 지시하지만 eval_068 처럼
**한쪽 진영 근거가 0건**인 경우까지는 막지 못했다(프롬프트 순응 실패). 사후 규칙으로
이중 방어한다.

- **핵심 함정**: "당시 여당/야당" 라벨은 **시점 기준**이지 정당 기준이 아니다.
  eval_019 는 인용 3건이 전부 "더불어민주당"(시점만 다름)인데 "여야 대립"으로
  서술됐다 — 정당명 자체가 2개 이상이어야 진짜 비교다.
- 규칙(순수 함수, 코드 스케치):

```python
def core_party(party_label: str | None) -> str | None:
    """'더불어민주당(당시 여당)' → '더불어민주당'. None(라벨 없음)·'정부측' 은 None.
    '무소속' 도 None — 어느 정당 진영도 대표하지 않으므로 비교 커버리지 집계에서
    제외한다 (무소속 1명 + 민주당 1명을 '정당 2개'로 세면 거짓 covered)."""

def comparison_coverage(cited_sources: list[dict]) -> dict:
    """cited_numbers 에 해당하는 sources 만 대상. core_party() 가 None 인
    발언자(정부측·증인·미상·무소속)는 집계에서 제외.
    반환: {"core_parties": {"더불어민주당"}, "covered": False}
    covered = len(core_parties) >= 2 (실제 서로 다른 정당명 ≥2)."""
```

- 질문이 `classify_question()` 에 `"compare"` 를 포함하는데 `covered is False` 이면
  **`verification_flags` 에 `"comparison_one_sided"` 추가** — grounding PARTIAL 강등.
- **1단계(즉시)**: 생성 **전** 신호로도 활용 — 검색된 sources 단계에서 이미 계산
  가능(citation 확정 전이라 "cited"가 아닌 "retrieved" 기준). `covered is False` 면
  `build_user_message` 에 진영별 실제 존재 정당명을 명시한 강한 지시를 추가:
  > "(안내: 근거에 등장하는 정당은 {실제 정당명 목록}뿐입니다. 다른 정당·진영의
  > 발언은 근거에 없으므로 비교하지 말고 '{없는 진영}의 입장은 이 회의록에서
  > 확인할 수 없습니다'라고 명시하세요. 같은 발언자를 서로 다른 진영으로
  > 서술하지 마세요.)"

  이건 새 LLM 호출이 아니라 **기존 호출의 프롬프트에 한 문단 추가**이므로 비용 0,
  지연 0.
- **사후 검증(§2 규칙)**은 이 사전 지시가 순응되지 않았을 때의 안전망 — 그래도
  `covered is False` 로 남으면 flag.

- **결정 필요 ②**: flag 가 서면 사용자 노출 텍스트를 어떻게 할지.
  - (a) 텍스트 불변, `grounding` 만 PARTIAL 강등 + 프론트 배지(UX 에이전트 영역)
  - (b) 규칙 위반 시 1회 재생성(§2-1 지시를 더 강하게, LLM 재호출 — 비용 2배지만
    희귀 경로이므로 일 예산 영향 작음, §5-2 참고)
  - (c) 코드가 결정적 문장을 답변 끝에 프로그래밍적으로 삽입("{없는 진영}의 입장은
    근거에서 확인되지 않았습니다.") — 답변 톤과 어긋날 위험, LLM 문장과 접합 부자연
  - 이 설계는 (a)를 1단계 기본값으로 제안하고 (b)를 2단계 옵션으로 열어둔다.
    최종 선택은 사용자 결정 필요.

### 2-2. 같은 발언자의 진영 이중 사용 (eval_068 의 "같은 위원인 김우영은…")

- 규칙: 답변 문장에서 화자명이 2회 이상 등장하고, 인접 문장이 서로 다른 진영
  프레이밍("여당은…", "반면 야당은…")에 각각 그 화자를 붙이면 flag
  `"speaker_both_sides"`. 화자명 등장 위치 + 주변 20자 내 "여당/야당/찬성/반대"
  대조 키워드 탐지로 충분 — 순수 정규식, LLM 불필요.
- 2-1 의 커버리지 가드가 이미 이 케이스를 대부분 포섭한다(같은 화자만 인용됐다는
  것은 core_parties 가 1개라는 뜻). 이 규칙은 **다른 화자 2명이 실은 같은 정당인
  경우**(2-1 로 커버) 와 **화자 1명을 코드가 억지로 양쪽에 배치한 경우**(이 규칙
  고유 커버리지)를 모두 잡기 위한 보조 신호.

## 3. 거짓 Q-A 짝짓기 감지 (eval_029)

- 표적 질의 패턴: 질문에 두 실명(또는 실명+직함)이 "질문자→답변자" 관계로 등장.
  `query_parser._ACTOR_RE` 를 확장하거나 신규 정규식으로 "A 위원이 B 장관에게 …
  질문했고 … 답변" 류를 탐지(정확한 정규식은 구현 단계 확정 — 이 설계는 신호
  존재만 규정).
- 검증 규칙: 답변이 인용 [n]으로 "질문"과 "답변"을 각각 구성할 때, 그 인용들의
  `committee` + `date` 가 **동일 회의**(같은 committee, 같은 date)를 가리키는지
  확인. 불일치하면 flag `"qa_pairing_date_mismatch"`.
  - eval_029 실측: 홍기원 질의 citation[1] = 외통위 2024-11-11, 조태열 답변으로
    쓰인 citation[5] = (근거 원문 재확인 결과) 2025-02 업무보고 — committee·date
    불일치 → 규칙이 정확히 잡는다.
- **결정 필요 ③**: 위반 시 처리. §2-1 의 (a)(b)(c) 옵션과 동일한 갈래 — 이 설계는
  "정직한 처리" 관점에서 (a)(텍스트 불변 + 강등 + 프론트 배지)를 1단계 기본값으로
  제안하되, 이 케이스는 프롬프트 사전 지시로도 예방 가능(§3 마지막 항 참고)하므로
  실사용 위반율이 낮을 것으로 예상 — 실측 후 재판단.
- **사전 지시(비용 0)**: `_ACTOR_RE` 매칭에 "질문…답변" 패턴이 잡히면
  `build_user_message` 에 "(안내: 질문자와 답변자가 다른 날짜의 발언이면 그 사실을
  명시하고 같은 회의의 질의-답변으로 단정하지 마세요.)" 추가 — 이것만으로도
  eval_029 류가 상당 부분 예방될 가능성.

#### 주어-인용번호 결속의 한계 (QA 지적 반영 — 정직한 처리)

문장 단위 규칙은 "이 문장의 주어(주장 주체)"와 "이 문장의 [n]"을 묶어야 하는데,
한국어 답변은 주어 생략이 흔하다. eval_013 실답변의 "…법안도 제안되었습니다[5]"가
정확히 이 케이스 — 문장 안에 명시 주어가 없고, 귀속 주체(외교부/정부)는 앞 문장에서
승계된다. 처리 방침:

- 1차(높은 정밀도): 문장 안에 **명시적 기관·화자명이 있을 때만** 화자/역할·정당
  라벨 규칙을 적용한다. 주어 생략 문장은 이 규칙들을 건너뛴다 — 재현율을 정직하게
  하향(오탐 0 우선, 배포 데모에서 잘못된 강등이 더 해롭다).
- 2차(재현율 보강, 같은 1단계 내 선택 구현): **문단 주어 승계 휴리스틱** — 같은
  문단에서 마지막으로 등장한 기관·화자명을 생략 주어로 간주해 규칙을 적용하되,
  승계 기반 flag 는 `detail` 에 `inherited_subject: true` 를 표기해 신뢰도를 구분.
  eval_013 은 이 휴리스틱이 있어야 잡힌다(§7 목표 수치는 2차 포함 기준).
- 그래도 안 잡히는 형태(주어도 승계도 모호)는 2단계 혼합 접근(§4-1 C)의 LLM 재확인
  후보로 넘긴다 — "의심" 판정 기준에 '주어 생략 + 귀속성 술어("제안했다"·"밝혔다")'
  조합을 포함.

### 4-3. eval_011 류 — "정부 입장 **과** 의회 반응" 다중 대상 정책요약

`classify_question()` 의 `"compare"` 는 "여야 비교" 전용이라 eval_011 같은
"A 는? B 는?" 이중 질문(정부 입장 + 의회 반응)에는 적용되지 않는다. 두 갈래:

- (i) `_COMPARE_RE` 를 넓혀 "…과/와 …은/는 어떠했나요" 류 이중 대상 질문 전반을
  잡고 §2 커버리지 가드를 재사용(간단하지만 오탐 위험 — "정당 비교"가 아닌 질문에
  가드 문구가 붙는 부작용)
- (ii) 별도 `"multi_target"` 유형 신설, 별도의 약한 지시("질문의 각 대상을 모두
  다뤘는지 확인하고, 못 다룬 대상은 명시적으로 부재를 공시하라")만 추가(진영
  커버리지 계산은 안 함 — 정당 무관 질문이라 부적절)

**결정 필요 ④**: (i) vs (ii) — 이 설계는 정확성상 (ii)를 권장하지만 구현 비용은
(i)이 더 낮다. 스코프 확정은 2단계 착수 전 재논의 권장(1단계 최소 스코프에는 (ii)
없이도 §2·§3·§4-2·§0-2 만으로 8건 중 7건 커버 가능 — §5-1·§7 목표 수치 참고).

## 5. 단계적 도입안

### 5-1. 1단계 — 저비용, 배포 즉시 가능 (추가 비용 $0, 지연 0~수 ms)

전부 **규칙 기반 + 프롬프트 사전 지시**. 신규 LLM 호출 없음.

1. `_COMPARE_RE` 확장(§0-2) — eval_057·068 이 기존 방어를 아예 못 받던 문제 해소.
2. 진영 커버리지 가드(§2-1) — 생성 전 신호 + 사후 규칙 이중 배치.
3. Q-A 짝짓기 사전 지시(§3 마지막 항).
4. 사후 규칙 4종(§2-2, §3 규칙, §4-2 화자/역할·키워드·정권기) → `verification.py` →
   `verification_flags` → `grounding.judge()` 강등 접합(§1).
5. `query_logs` 에 `verification` 컬럼(JSON) 신설해 flag 이력 축적(POL-8 의
   `issue_context` 저장 패턴과 동일 — 컬럼 추가 필요, 스키마 마이그레이션 1개).

- **비용/지연 정합성**: 규칙 연산은 서버 CPU 이므로 `DAILY_COST_LIMIT_USD=1.0`·
  `RATE_LIMIT_LLM_PER_MIN=5` 어느 것도 건드리지 않는다. 지연도 사실상 무증가라
  기존 qa 2.5~13초/report 10~18초 유지.
- **예상 커버리지**: §0 표 기준 **8건 중 7건**이 1단계만으로 최소 flag 발생
  (grounding 강등 + 로그)까지 달성 —
  · eval_019: 질의가 이미 `_COMPARE_RE` 에 매칭되므로("비교" 리터럴) 정규식 확장과
    무관하게 §2-1 사후규칙이 **직접** 잡는다(인용 정당이 더불어민주당뿐 →
    covered=False). 초안의 "간접 개선" 서술은 부정확했던 것 — 정정.
  · eval_057: §4-2 정당 라벨 일치(§0-1) + 키워드 포함률 — 이제 "부분"이 아니라 직접.
  · eval_068·029·013(§4-2 주어 승계 휴리스틱 전제)·055·035: §0 표의 각 규칙.
  · **eval_011 은 1단계 미커버** — §4-3 결정 필요 ④ 확정 후 커버(아래 §7 목표와 정합).
  텍스트 자체가 고쳐지는지는 §2-1/§3 의 "결정 필요" 선택에 따라 갈린다.

### 5-2. 2단계 — 품질 최대 (혼합 접근, 선별적 LLM 비용)

1. §4-1 접근 C(혼합) 도입 — 1단계 규칙이 "의심" 판정한 문장만 대상.
2. §2-1/§3 위반 시 옵션 (b)(1회 재생성) 채택 여부 결정 후 구현 — 위반율이 낮으면
   (§5-1 이후 실측) 비용 영향 미미.
3. eval_011 류 다중 대상 부재공시(§4-3)를 (ii) 방식으로 정식 구현.
4. 비교 모델 승급 검토 — 혼합 접근의 LLM 단계만 gpt-4o 로 승급해 정확도 재측정
   (비용 5~10배지만 트리거율이 낮아 총액 영향 작을 가능성 — 실측 필요).

- **비용/지연 정합성**: 트리거율 20~30% 가정 시 평균 추가 비용 질의당
  $0.00005~0.0002(§4-1 표) — 현재 실측 기준선(qa $0.0005) 대비 10~40% 증가.
  `DAILY_COST_LIMIT_USD=1.0` 여유 안에서 흡수 가능(현재 배포 규모가 일 수십~수백
  질의 수준이라 판단 — 실사용 트래픽 실측 전엔 추정치). 지연은 트리거된 질의에서만
  qa +1~2초 — **report 모드는 이미 10~18초라 2단계 지연 추가에 더 민감**, report
  우선순위를 낮추거나 비동기(§5-3)로 분리하는 편이 안전.

### 5-3. (2단계 대안, 결정 필요 ⑤) 동기 vs 비동기

- 동기: 응답 자체가 검증 통과 후 반환 — 사용자는 항상 "검증된" 답변을 보지만
  지연이 늘어난다.
- 비동기: 답변은 즉시 반환하고, 검증은 백그라운드에서 실행해 `query_logs` 에만
  기록(모니터링/차기 eval 재료) — 사용자 응답 지연 0이지만 그 답변 자체는 실시간
  교정되지 않는다(피드백 루프 성격).
- 이 설계는 배포 규모(포트폴리오 데모, 저트래픽)를 고려해 **1단계는 동기(규칙은
  어차피 무비용무지연이라 선택의 의미가 없음), 2단계는 비동기를 기본 제안**하되
  최종 결정은 사용자 몫으로 남긴다.

## 6. `verification.py` 인터페이스 스케치 (설계 수준, 구현 아님)

```python
# backend/verification.py (신규, POL-8 issue_context.py 패턴)

def core_party(party_label: str | None) -> str | None: ...
def comparison_coverage(cited_sources: list[dict]) -> dict: ...  # §2-1
def speaker_both_sides(answer: str, cited_sources: list[dict]) -> bool: ...  # §2-2
def qa_pairing_dates(answer: str, cited_sources: list[dict], question: str) -> bool: ...  # §3
def speaker_role_consistency(answer: str, cited_sources: list[dict]) -> list[str]: ...  # §4-2
def party_label_consistency(answer: str, cited_sources: list[dict]) -> list[str]: ...  # §4-2 (eval_057 — 답변 "화자명(정당명)" vs 주입 party 라벨)
def keyword_containment(answer: str, cited_sources: list[dict]) -> list[str]: ...  # §4-2
def ruling_period_consistency(answer: str, cited_sources: list[dict]) -> list[str]: ...  # §4-2 (party.RULING_PERIODS 재사용)

def verify(question: str, result: dict) -> dict:
    """generate_answer() 반환 dict 를 받아 규칙 전부 실행 → 
    {"flags": [...], "detail": {...}} 반환. 규칙 기반이라 예외 시에도
    빈 flags 로 안전 강등(생성 실패로 번지지 않게 — issue_context 의 예외 처리 패턴)."""
```

- **호출 위치 = `answer.py` 내부로 확정** (초안의 "결정 필요 ⑥" 철회 — QA 지적
  반영, 이건 데이터 접근 제약이 정하는 사안이지 취향이 아니다). 근거:
  `generate_answer()` 반환 dict 의 `sources`/`citations` 는 `_source_summary()`
  (answer.py:408-415)를 거치며 **`text` 를 200자 snippet 으로 절단**한다. 근거 턴
  전문(EVIDENCE_TURN_MAX 4,000자)은 `generate_answer` 내부 지역변수 `sources` 로만
  존재하므로, `main.py` 에서 `judge()` 옆에 호출하면 `keyword_containment` 등
  본문 대조 규칙의 재현율이 붕괴한다(200자 안에 핵심 키워드가 없으면 전부 오탐).
  따라서 `generate_answer()` 가 LLM 응답 수신 직후, `_source_summary` 절단 **전**의
  전문 sources 로 `verify()` 를 호출하고 반환 dict 에 `verification` 키를 싣는다.
  `main.py` 는 그 값을 읽어 grounding 강등(§1)과 query_logs 적재만 한다.

   기준"으로 삼는다(에러 유형 텍스트가 이미 있어 재현 여부를 사람이 빠르게
   대조 가능).
2. **재실행 절차**: 1단계 구현 후 이 8건의 질의만 `generate_answer()` 재실행
   (검색 결과가 코퍼스 불변이면 동일 — 변경 없으면 `hits` 캐시 재사용 가능) →
   `verification_flags` 발생 여부 + 답변 텍스트 변화 확인 → 사람이 근거 원문 대조로
   재판정(기존 4단계 사람검수와 같은 방식).
3. **목표 수치** (§5-1 커버리지와 정합 — 초안의 8/8 은 §4-3 미커버 명시와 모순,
   정정):
   - 1단계 직후: **탐지율 목표 7/8** (eval_011 제외 — §4-3 다중 대상 부재공시는
     결정 필요 ④ 확정 후 별도 커버). eval_013 은 주어 승계 휴리스틱(§4-2 2차)
     포함 기준 — 1차 규칙만 구현하면 6/8 로 하향.
   - 결정 ④ 확정·구현 후: 8/8.
   - 최종(2단계 포함) grading 목표: 8건 모두 `overall: pass` **또는**
     `human_notes` 가 "부분 답변+부재 공시 = 모범 처리"(2026-07-15 기록의
     eval_016·075 선례) 패턴으로 재분류 — 심판 채점이 이 패턴을 여전히 fail 로
     오판정할 수 있으므로 사람 재확인 필수(기존 교훈 — "자동채점 fail 은 방향
     예측 불가").
4. **전체 회귀 무손상**: 75문항 전체를 `scripts/answer_eval_judge.py` +
   `answer_eval_score.py` 로 재측정 — 확정 기준선 **67/75 = 89.3%** 이상 유지
   (하락 시 사전 지시문·규칙 임계값이 다른 유형에 부작용을 낸 것 — 원인 분리 필요).
5. **비용·지연 실측**: `query_logs.usage.est_cost_usd`/`latency_ms` 를 1단계
   배포 후 실 사용 1주 집계 → 2단계 트리거율 가정치(§5-2 의 20~30%)를 실측치로
   교체 → `DAILY_COST_LIMIT_USD` 여유 재확인.

## 8. 범위 밖 (이번 설계에 포함 안 함)

- grounding.py 의 4단계 등급 체계 자체의 재설계(신규 등급 신설) — §1 에서 기존
  4단계 유지로 명시 확정.
- LLM judge(`scripts/answer_eval_judge.py`, gpt-4o) 자체의 정확도 개선 — 측정
  도구는 그대로 재사용.
- 검색층(하이브리드·리랭커) 개선 — eval_019 의 "근거 5건 전부 무관"은 검색 실패가
  근본 원인이고, 이 설계의 §2 커버리지 가드는 **생성 단계의 정직한 처리**만
  보장한다(검색이 애초에 관련 근거를 못 찾으면 "확인할 수 없다"로 유도할 뿐,
  검색 품질 자체는 백로그 "⑦리랭커 절차발언 디부스트"의 몫).
- 프론트엔드 UX(검증 flag 배지·경고 표시) — UX/UI 에이전트 영역, 이 스펙은
  `verification` 필드의 존재와 스키마까지만 규정.
- query_logs 스키마 마이그레이션 실행(§5-1 항목 5) — 다음 바퀴 구현 시 처리.
- 비동기 검증 파이프라인의 큐·재시도 인프라(§5-3 채택 시 필요) — 설계만 언급,
  세부 구현은 2단계 착수 시.
