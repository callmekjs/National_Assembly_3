## 평가 전제

이 프로젝트는 **LLM 자체(파운데이션 모델)가 아니라 LLM 기반 응용 서비스**다 —
GPT-4o-mini(답변)와 text-embedding-3-small(검색)이라는 기성 모델 위에
검색·근거·신뢰 장치를 얹은 RAG 시스템. 따라서 Making_LLM.md **Part 1(모델 제작)
기준은 해당 없음**이고, Part 2(서비스) 10가지 기준으로 평가한다.
이는 감점 사항이 아니다 — 개인 프로젝트에서 ②를 선택한 것 자체가 기준 1을 통과한 판단이다.

## 종합 평가: **B+ ~ A- (개인/포트폴리오 프로젝트 기준 상위권)**

한 줄 요약: **"모델을 믿지 않는 구조"와 "숫자로 관리하는 문화"라는, 상용 LLM 서비스도
자주 빠뜨리는 두 가지를 갖췄다.** 약점은 LLM 응용 특유의 것이 아니라 일반 서비스
운영 성숙도(배포 준비, 재실행 안전성, API 계층 테스트)에 몰려 있다.

## 기준별 상세

### 1. 문제 정의 — A

- LLM 에 맡긴 것: 자연어 질문 이해, 여러 근거의 요약·종합 (LLM 없이는 불가능한 부분).
- LLM 에서 뺏은 것: 정당·여야 라벨(코드가 members 테이블로 판정해 주입), 한자 이름
  병기(별칭 사전), 인용 번호 검증(정규식). **"모델이 추측할 수 있는 것은 추측할 필요가
  없게 만든다"** — 이 원칙이 코드 전반에 일관되게 적용되어 있고, 이는 연차 높은
  LLM 엔지니어의 설계 습관이다.

### 2. 지식 주입 — A

- RAG 선택이 정당한 문제(출처 제시가 서비스의 존재 이유인 GovTech, 회의록은 계속 추가됨).
- 검색 설계의 모든 결정에 실측 근거가 붙어 있다:
  - 벡터 단독의 고유명사 실패("티메프" 528청크 존재하는데 못 찾음)를 **측정한 뒤**
    하이브리드(RRF) 채택 — 유행 따라 하이브리드를 쓴 게 아니라 필요를 증명하고 씀
  - 한국어 검색은 FTS 가 조사 붙은 형태를 21% 놓치는 것을 실측하고 pg_trgm 채택
  - 키워드 가중치 1.2 도 baseline 저장 → 7문항 비교 후 적용
- 청킹(2,500자+문장 분할)은 무난하나 한국어 구두점 소실 케이스 보강 여지 (fix_checklist 2순위).

### 3. 환각 억제 — A-

교과서적인 다층 방어가 구현되어 있다:

| 층 | 구현 | 효과 |
|---|---|---|
| 사전 차단 | 검색 0건→NONE, 유사도 미달+키워드 0건→REFUSED (LLM 미호출) | 근거 없는 질문에 비용 0 으로 거절 |
| 프롬프트 | 근거 블록 밖 서술 금지 + 여야 질문 안내문(질문 인접 배치) | 1차 방어 |
| 사후 판정 | [n] 인용 파싱·검증, 거절 문구 감지(변형 3종 실측 반영), 무인용 주장→ungrounded 경고 | 순응 실패를 코드가 잡음 |
| 코드 주입 | 정당 라벨·발언 자격(role) 게이트 — 의원만 정당, 행정부는 "정부측", 후보자·증인 무표기 | 실측된 오분류(겸직 장관 여당 표기) 재발 차단 |
| 후처리 | strip_boilerplate — 금지 상투구를 결정론적으로 제거 | "프롬프트 순응은 확률적" 인식 |

- 특히 "여야별 정리" 질문에서 LLM 이 정당을 추측 생성하던 문제를 프롬프트 →
  안내문 → 코드 주입 3단계로 몰아간 과정은 이 분야 모범 사례 수준.
- 감점: role=NULL 발언자(증인 등)가 동명 의원과 겹치면 정당 라벨이 잘못 붙을 수 있는
  구멍 1개 (2026-07-06 검토 발견, fix_checklist 6순위 — 이 모듈이 막으려는 바로 그 실패).

### 4. 평가(eval) — B+

- **검색 평가는 모범적**: 63문항 13유형 정답셋, criteria 방식(데이터 재처리에도 유효),
  unanswerable 4문항 반전 채점(거절 능력 측정), 변경마다 재실행하는 습관.
  eval 주도 개선의 실증 사례까지 있다: date_based 0.00 발견 → query_parser 신설 →
  1.00, 전체 R@5 0.746→**0.949**.
- **답변 평가는 부재**: Recall@5=0.949 는 "근거를 찾아왔는가"까지만 말해준다.
  "LLM 이 근거를 올바르게 해석했는가, 여야를 안 틀렸는가"는 스모크(수동 확인)뿐,
  정답셋·지표가 없다. 기준 4가 경고하는 전형적 상태 — 검색 층만 있고 답변 층이 없음.
  (POL-7 로 계획은 잡혀 있고, 07-06 추가된 source_block 로그가 그 재료가 된다.)

### 5. 컨텍스트 설계 — A-

- 근거 블록: 번호+발언자+자격+정당라벨+위원회+날짜 구조화, chunk_id 는 LLM 미노출
  (번호↔ID 매핑은 코드 보관), snippet 대신 전문 재조회.
- 복수 위원회 질문은 근거를 위원회별 섹션으로 그룹핑 — 모델이 출처 위원회를 착각해
  다른 문단에 배치하던 실측 오류를 **구조로** 제거 (기준 5의 정석).
- 긴 발언(9조각 2만 자)이 조각 하나만 들어가던 맥락 절단은 07-06 수정 완료
  (turn 전문 복원, 상한 4,000자 + 적중 조각 중심 창).
- report 모드의 인접 턴 보조 맥락(500자 절단, "인용 근거로는 본문만" 명시)도 토큰
  예산과 맥락의 균형을 아는 설계.

### 6. 비용·지연 — B+

- 원가를 아는 프로젝트: 임베딩 총 ~$2.4, qa ~$0.0005/질문, report ~$0.0017/질문,
  latency_ms 를 응답·로그에 기록. 사전 차단으로 무의미한 LLM 호출 자체를 제거.
- 임베딩 배치·증분 처리·지수 백오프·--dry-run 비용 예측 — 대량 호출의 기본기 충족.
- 감점 2건 (fix_checklist 4순위): 재시도가 base APIError 를 잡아 **영구 오류(401/400)도
  6회 재시도**, 청커의 무제한 조각이 8,192토큰 초과로 배치 전체를 죽일 수 있음.

### 7. 로깅·관측성 — A-

- query_logs: 질문·답변·grounding 등급·인용·비용·지연·피드백(rating)까지 한 행에.
  거절 문구 신종 변형을 "PARTIAL+ungrounded 행"에서 발굴하는 운영 루프도 정의되어 있다.
- 07-06 보강: **source_block(LLM 에 실제 들어간 근거) 저장** — "왜 이런 답이 나왔지?"를
  재현할 수 있게 됨. 기준 7의 핵심 요건 충족 (이전까지는 이 부분이 구멍이었다).
- 감점: 로그 실패가 print 로만 남음(모니터링 연결 없음) — 규모가 커지면 보강 필요.

### 8. 안전 — C+

- 확보된 것: 입력 검증(날짜 타입, rating 1~5, 질문 2~1,000자 — 07-06), SQL 전면
  파라미터 바인딩(인젝션 없음), XSS 없음(react-markdown, dangerouslySetInnerHTML 미사용),
  비밀키 .env 분리·gitignore. 데이터는 공개 회의록이라 개인정보 부담 자체가 낮고,
  원본의 익명화 표기(박00)도 그대로 보존.
- 없는 것: **rate limit·인증** (질문 1건 = LLM 비용이므로 공개 배포 시 비용 공격에 무방비),
  CORS·API 주소가 개발 환경 고정. 로컬 데모 단계라 치명적이지 않지만 4단계(배포) 전
  필수 — 현재는 "배포하지 않았기 때문에 안전"한 상태.

### 9. 테스트 — B+

- 순수 로직 커버리지가 좋다: 인용 파서, grounding 판정표(변형 문구 회귀), 정당
  라벨(여야 경계일·NFKC 한자·동명이인·위성정당·자격 게이트), 파서 실측 헤더 50케이스,
  "게이트의 게이트"(품질 게이트 자체가 고장나는 사고의 재발 방지) — 전부 LLM·DB 없이 돈다.
- 07-06 정합화: pytest 에서 무조건 통과하던 함정 제거(assert 화), 그 즉시 전역 상태
  오염 버그가 드러나 수정 — "실패할 수 있는 테스트" 기준을 이제 충족.
- 감점: HTTP API 계층(TestClient) 0건, 프론트 0건 — 로직과 사용자 사이 이음새가 무검증.

### 10. 데이터 파이프라인 — B

- 강점: 단계별 산출물 버전 관리, 품질 게이트 2종(BLOCK/WARNING), **원본 대조 감사
  300/300**, 버려진 헤더 리포트로 "조용한 폐기" 가시화 — 그 리포트 덕에 6.2만 턴
  유실(인사청문회·계엄 국면 발언 통째 누락)을 발견·복구한 실적이 있다. 전 스크립트
  UTF-8 명시(Windows cp949 함정 전무)도 드물게 깔끔한 부분.
- 약점 (fix_checklist 4순위, 미수정): 쓰기가 비원자적 + "파일 존재=완료" 스킵이라
  중단 시 반쪽 산출물이 영구 고착, 크롤러가 덮어쓴 정정본 PDF 를 추출기가 무시
  (SHA-256 을 계산만 하고 안 씀), 소스별 실패가 exit 0 으로 삼켜짐. 임베딩 CASCADE
  유실은 07-06 수정 완료. 기준 10이 경고하는 "조용한 유실" 위험이 남은 영역.

## 총평

**잘한 것의 본질**: 이 프로젝트의 등급을 끌어올린 것은 코드량이 아니라 두 가지 태도다.
① 모델의 순응을 믿지 않고 구조(사전차단·사후판정·코드 주입·후처리)로 막는다.
② 결정을 감이 아니라 실측(eval 점수, 실패 사례 재현)으로 한다.
이 둘은 Making_LLM.md 기준 3·4의 핵심이자, 실무에서도 가장 자주 무너지는 지점이다.

**남은 격차**: LLM 특유의 영역이 아니라 서비스 성숙도 — 답변 층 평가(기준 4),
파이프라인 재실행 안전성(기준 10), 배포 안전장치(기준 8). 셋 다 `docs/fix_checklist.md`
에 항목화되어 있어 "몰라서 못 한 것"이 아니라 "아직 안 한 것" 상태다.

**권고 우선순위**:
1. **답변 평가셋 구축** (B+ → A 로 가는 최단 경로): source_block 로그가 쌓이는 중이므로,
   실질의 30~50건에 수동 라벨(근거 해석 정확/여야 분류 정확/인용 적절)을 붙여 기준선 확보.
   POL-7 과 통합 가능.
2. **ETL 재실행 안전성** (tmp 파일+os.replace, SHA-256 스킵 판정, 실패 목록): 다음
   파이프라인 재실행 전에 — 지금 고치지 않으면 다음 재처리 때 조용한 유실 위험.
   → **2026-07-07 완료 (아래 재평가 참조)**
3. **배포 전 안전장치** (rate limit, CORS/주소 설정화, 인덱스 생성 SQL 저장소 반입):
   4단계 착수 조건으로.
