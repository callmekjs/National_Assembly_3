# LLM을 만든다는 것 — 기준과 체크리스트

> 작성: 2026-07-06. "LLM을 만들려면 어떤 기준이 있어야 하는가"에 대한 정리.
> 함께 볼 문서: `docs/llm_comparison_report.md` (이 기준으로 본 프로젝트를 평가한 보고서)

## 0. 먼저 — "LLM을 만든다"의 두 가지 의미

| | ① 모델 자체를 만들기 (파운데이션 모델) | ② LLM 으로 서비스를 만들기 (응용) |
|---|---|---|
| 예 | GPT, Claude, Llama, HyperCLOVA | ChatGPT 래퍼, RAG 검색봇, **이 프로젝트** |
| 주체 | OpenAI·Anthropic·네이버 등 소수 기업 | 개발자·스타트업 대부분 |
| 비용 | 수백억~수천억 원 (GPU + 데이터 + 인력) | 월 수천 원~수백만 원 (API 호출료) |
| 기간 | 년 단위 | 주~월 단위 |

세상의 "LLM 프로젝트" 99% 는 ②다. 이 프로젝트(국회 회의록 RAG)도 ② —
**GPT-4o-mini 라는 기성 LLM 위에 검색·근거·신뢰 장치를 얹은 응용 서비스**다.
"LLM 을 만들었다"보다 "LLM 기반 서비스를 만들었다"가 정확한 표현이고,
평가 기준도 ①이 아니라 ②의 기준을 적용해야 한다.

## Part 1 — 모델 자체를 만드는 기준 (참고용)

직접 만들 일은 거의 없지만, ②를 잘하려면 ①이 무엇으로 이루어지는지는 알아야 한다.

1. **데이터** — 수조 토큰 규모의 텍스트. 수집보다 어려운 것이 정제(중복 제거,
   품질 필터, 개인정보 제거, 라이선스 검토). "모델 품질 = 데이터 품질"이 제1원칙.
2. **컴퓨트** — 수천~수만 장의 GPU 를 수개월. 학습 중단·재개(체크포인트), 손실 발산
   대응 같은 운영 기술이 별도로 필요.
3. **아키텍처와 토크나이저** — 트랜스포머 변형 선택, 파라미터 수·컨텍스트 길이 결정.
   토크나이저는 언어 특성(한국어 교착어 등)에 큰 영향.
4. **사전학습 → 정렬(alignment)** — 다음 단어 예측으로 사전학습한 뒤, 지시를 따르게
   만드는 SFT(지도 미세조정)와 사람 선호를 반영하는 RLHF/DPO 단계.
5. **평가** — 벤치마크(MMLU 등) + 실사용 평가. 학습 데이터에 벤치마크가 새어 들어가는
   오염(contamination) 관리.
6. **안전** — 유해 출력 억제, 탈옥(jailbreak) 방어, 편향 측정. 별도 팀이 붙는 영역.

**결론: 이 길은 기준이 "얼마나 잘"이 아니라 "할 수 있는가"부터 시작한다.**
개인·소규모 팀의 선택지는 사실상 ②이고, 그건 열등한 길이 아니라 합리적인 길이다.

## Part 2 — LLM 기반 서비스를 만드는 기준 (본론)

20년 넘게 이 바닥에서 반복해서 확인한 것: **서비스의 품질은 모델이 아니라
모델을 둘러싼 시스템이 결정한다.** 같은 GPT-4o-mini 를 써도 아래 10가지를
갖춘 서비스와 아닌 서비스는 완전히 다른 물건이 된다.

### 기준 1. 문제 정의 — "LLM 이 왜 필요한가"에 답할 수 있는가
- 검색으로 충분한 문제에 LLM 을 쓰면 비용·환각만 얻는다.
- 좋은 프로젝트는 "LLM 없이는 안 되는 지점"(자연어 질문 이해, 여러 근거의 요약·종합)과
  "LLM 에 맡기면 안 되는 지점"(사실 판정, 분류 라벨)을 구분한다.

### 기준 2. 지식 주입 — RAG vs 파인튜닝을 근거 있게 선택했는가
- 사실이 계속 갱신되고 출처 제시가 필요하면 **RAG** (검색해서 근거를 주고 답하게).
- 말투·형식·도메인 습관을 바꾸려면 파인튜닝. 사실 주입 목적의 파인튜닝은 대부분 실패한다.
- RAG 를 골랐다면 검색 품질이 곧 답변 품질 — 청킹 전략, 하이브리드 검색(키워드+벡터),
  임베딩 모델 선택이 핵심 결정이 된다.

### 기준 3. 환각 억제 — "모델을 믿지 않는" 구조가 있는가
- 프롬프트로 "지어내지 마"라고 부탁하는 것은 방어가 아니다. 구조로 막아야 한다:
  - 근거에 없는 답은 거절하게 하는 **사전 차단** (검색 결과가 부실하면 LLM 호출 자체를 안 함)
  - 답변의 인용을 코드로 검증하는 **사후 판정** (인용 번호 유효성, 근거-답변 대조)
  - LLM 이 추측하기 쉬운 정보(소속·날짜·수치)는 **코드가 주입** (모델의 일이 아니게)
- 확인 못 하면 "모른다"고 답하는 것이 정답인 설계 — 거절도 품질이다.

### 기준 4. 평가(eval) — 감이 아니라 숫자로 관리하는가
- "좋아진 것 같다"는 개선이 아니다. 정답셋 + 지표(Recall@k, MRR 등) + 변경 때마다 재측정.
- 평가는 두 층이다: **검색 평가**(정답 근거를 찾아오는가)와 **답변 평가**(LLM 이 근거를
  올바르게 해석·인용하는가). 전자만 있고 후자가 없는 프로젝트가 많다 — 후자가 더 어렵다.
- 답할 수 없는 질문(unanswerable) 세트를 반드시 포함 — 거절 능력을 측정해야 한다.

### 기준 5. 프롬프트·컨텍스트 설계 — 모델에게 무엇을 어떻게 보여주는가
- 근거를 그냥 쌓지 말 것: 번호·메타데이터(발언자·날짜·출처) 있는 구조화된 블록,
  긴 원문의 맥락 보존, 토큰 예산 관리.
- 프롬프트 순응은 확률적이다 — 중요한 규칙은 프롬프트 + 후처리(코드) 이중으로.
- 모델이 헷갈릴 판단은 구조로 제거 (예: 출처를 섹션으로 묶어 배치 오류 원천 차단).

### 기준 6. 비용·지연 — 호출당 원가를 아는가
- 질문 1건당 비용(토큰 단가 × 사용량)과 응답 시간을 실측·기록. 모르면 확장 못 한다.
- 불필요한 호출 제거(캐시, 사전 차단), 재시도 정책(일시 오류만, 지수 백오프),
  영구 오류(잘못된 키, 한도 초과)는 즉시 실패.

### 기준 7. 로깅·관측성 — "왜 이런 답이 나왔지?"를 재현할 수 있는가
- 질문·답변만 로그하면 부족하다. **모델에 실제로 들어간 프롬프트/근거**를 남겨야
  이상 답변을 사후 분석할 수 있다.
- 로그는 답변 품질 평가셋(기준 4)의 재료가 된다 — 로깅과 평가는 한 몸.

### 기준 8. 안전·개인정보·비용 방어
- 입력 검증(길이 제한, 형식 검증) — 악의적/실수 입력이 비용 폭탄·서버 다운으로 이어지지 않게.
- 원문 데이터의 개인정보·저작권 검토. 시스템 프롬프트 유출, 프롬프트 주입 대응.
- 공개 서비스라면 rate limit(호출 횟수 제한)과 인증은 선택이 아니라 필수.

### 기준 9. 테스트 — LLM 없이 돌아가는 테스트가 있는가
- LLM 호출 없이 검증 가능한 순수 로직(파서, 판정 규칙, 조립)을 최대화하고 단위 테스트.
- LLM 출력은 비결정적 — 정확 일치 대신 성질 검사(인용이 유효한가, 거절 문구가 있는가).
- 테스트가 "실패할 수 있는지" 확인할 것 — 무조건 통과하는 테스트는 없는 것보다 나쁘다.

### 기준 10. 데이터 파이프라인 — 원천 데이터를 신뢰할 수 있는가
- RAG 의 바닥은 ETL 이다: 원본 → 추출 → 정제 → 청킹 → 임베딩 각 단계에
  품질 게이트(불량이면 다음 단계로 못 가게)와 원본 대조 감사.
- 재실행 안전성: 중단·재개, 원본 갱신 반영, 실패 목록 기록. "조용한 유실"이 최악의 적 —
  없는 게 아니라 있는데 못 찾는 상태는 사용자가 눈치챌 수 없다.

## 요약 체크리스트

| # | 기준 | 핵심 질문 |
|---|------|----------|
| 1 | 문제 정의 | LLM 이 꼭 필요한 지점과 맡기면 안 되는 지점을 구분했는가 |
| 2 | 지식 주입 | RAG/파인튜닝 선택에 근거가 있는가, 검색 품질을 챙겼는가 |
| 3 | 환각 억제 | 프롬프트 부탁이 아니라 구조(사전차단·사후판정·코드 주입)로 막는가 |
| 4 | 평가 | 정답셋과 지표가 있고 변경마다 재측정하는가 (검색+답변 두 층) |
| 5 | 컨텍스트 설계 | 근거 블록이 구조화되어 있고 중요한 규칙은 코드로 이중화했는가 |
| 6 | 비용·지연 | 호출당 원가를 실측하는가, 재시도가 영구 오류를 구분하는가 |
| 7 | 로깅 | 모델에 들어간 것을 저장해 이상 답변을 재현할 수 있는가 |
| 8 | 안전 | 입력 검증·rate limit·개인정보 검토가 있는가 |
| 9 | 테스트 | LLM 없이 도는 테스트가 있고, 그 테스트가 실패할 수 있는가 |
| 10 | 파이프라인 | 품질 게이트·원본 감사·재실행 안전성이 있는가 |
